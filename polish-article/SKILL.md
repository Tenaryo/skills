# Polish Article

逐段审阅用户指定的文本文件，对排版、口语化表达、语义不清之处提出修改建议，每一处修改都需要用户确认后才执行。

## Core Principles

- **逐段审阅** — 按段落顺序检查，不遗漏任何内容
- **每改必问** — 每一处拟修改都必须获得用户批准
- **一次一处** — 不要一次抛出多个修改点，逐个确认
- **充分上下文** — 提问时展示原文和修改理由，让用户无需猜测
- **尊重原意** — 修改以忠实原文含义为前提，不添加、不删减核心信息

## What to Polish

扫描每一处文本，关注以下三类问题：

### 1. 排版问题
- 标题层级不统一（如混用 `#` 和 `##`）
- 列表格式不一致
- 段落间距不合理
- 缺少或多余的空行
- 中英文混排时缺少空格（如 `使用Python开发` → `使用 Python 开发`）
- 代码块缺少语言标注
- 标点符号混用（中英文标点不一致）

### 2. 口语化表达
- 过于随意的措辞（如 "然后就是说" "反正" "说白了"）
- 不正式的语气词
- 可替换为更书面化的表达
- 重复冗余的表述

### 3. 表达不清晰
- 句子过长、嵌套过深，难以理解
- 指代不明（"这个""那个"等）
- 逻辑跳跃、缺少过渡
- 术语使用前后不一致
- 含义模糊、可作多种理解的表述

## Workflow

```
1. 读取文件 → 2. 逐段扫描 → 3. 发现问题 → 4. 提问确认 → 5. 执行修改
                                    ↑                          ↓
                                    ← 6. 继续下一段 ←─────────┘
```

### Phase 1: 读取与全局预览

1. 读取用户指定的文件
2. 快速浏览全文，了解文章的整体结构和主题
3. 识别文章语言（中文 / 英文 / 混合），据此选择对应语言的修改风格
4. 向用户简要汇报文章概况（段落数、大致主题、初步印象），确认准备开始逐段审阅

### Phase 2: 逐段审阅

对每个段落执行以下步骤：

1. **阅读段落** — 理解该段的核心含义
2. **检测问题** — 按"排版 → 口语化 → 表达不清晰"的优先级扫描
3. **发现问题** → 进入 Phase 3
4. **无问题** → 告知用户该段无需修改，继续下一段

### Phase 3: 逐项提问确认

对于段落中发现的每个问题：

**⚠️ 关键约束：每次只提一个问题，等待用户回答后再提下一个。**

提问格式：

```
【问题类型】排版问题 / 口语化表达 / 表达不清晰

【原文】
> 展示原文片段

【修改建议】
> 展示修改后的文本

【修改理由】
简要说明为什么要这样改

【是否批准此修改？】
```

使用 question 工具提问，正文中展示原文、建议和理由，question 工具中只放简短的确认问题。

可能的用户回复：
- **批准** → 执行修改，继续下一个问题
- **拒绝** → 跳过该修改，继续下一个问题
- **提出替代方案** → 按用户方案修改，继续下一个问题

### Phase 4: 执行修改

所有问题确认完毕后：

1. 汇总所有被批准的修改
2. 一次性应用到文件中
3. 向用户汇报修改摘要（共修改了多少处，涉及哪些类型）

### Phase 5: 完成汇报

向用户提供修改总结：

1. **修改统计** — 排版修复 X 处，口语化修改 Y 处，表达优化 Z 处
2. **主要改动** — 列出最重要的几处修改及其前后对比
3. **剩余建议** — 如果有用户拒绝的修改，可选择性提示（用户可随时要求重新审视）

## Asking Guidelines

遵循 grill-me 的询问原则：

- **宁可多问，不要少问** — 不确定是否需要修改时，先问用户
- **不要替用户做决定** — 提出建议但不强制
- **尽早问** — 发现问题立刻提出
- **任何时间问** — 遇到不确定的情况随时停下来问

当遇到以下情况，必须停下来询问：
- 不确定某处是否算"问题"（边界情况）
- 修改可能改变原文含义
- 多种修改方案各有优劣
- 不理解原文想表达什么

## Edge Cases

- **文件很长** — 告知用户总段落数，询问是否需要跳过某些部分
- **多语言混合** — 分别处理，但注意中英文排版规范不同
- **代码与文本混合** — 只处理文本部分，不修改代码内容
- **用户中途要求停止** — 立即停止，汇报已完成的修改
